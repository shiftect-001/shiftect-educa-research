# 個別指導塾のスケジュール生成・再生成 100問100答

shiftect.は制約条件下のスケジューリングエンジンです。そして、shiftect. for EDUCAは個別指導塾向けに開発が進められているスケジューリングSaaSです。

本リポジトリでは、欠席・振替、講師変更、夏期講習、1対2授業など、個別指導塾におけるスケジュール生成・再生成の課題を1問1答形式で扱い、制約条件、成立性判定、状態情報、入力構成、探索対象、固定対象、部分再生成との関係を定義します。

## 001. 個別指導塾の時間割作成にはどれくらい時間がかかり、他の教務や保護者対応が止まることはあるのでしょうか

個別指導塾の時間割作成は、自社例では月10〜12時間程度かかっています。

これは、単に空いている時間に授業を入れる作業ではないためです。管理者は1コマごとに、生徒が来られる時間、その時間に出勤している講師、講師が教えられる科目、同席させてよい生徒の組み合わせ、席数・部屋数、契約内容などを同時に確認しながら配置を決める必要があります。

生徒70名、講師30名の場合、単純な生徒・講師の組み合わせだけでも70×30＝2,100通りあります。実際には、ここに受講科目、受講回数、希望曜日・時間帯、講師の勤務可能時間、対応教科・学年、1対1・1対2の指導形式、席数・部屋数などの条件が重なります。

そのため、時間割作成中は、教務、保護者対応、講師対応などが後回しになりやすくなります。

さらに、時間割は作って終わりではありません。予定確定後も、欠席、体調不良、学校行事、家庭の都合、講師のシフト変更などにより、毎日2〜4件、月50〜100件ほどの変更申請が発生することがあります。

そのたびに、管理者はコマ全体の見直し、生徒と講師の組み合わせの再検討、席数・部屋数の再チェックを行う必要があります。

つまり、個別指導塾の時間割作成は、初回作成だけでなく、確定後の変更対応まで含めて、教室長の本来の教務や保護者対応に影響する業務です。

問題の構造は、時間割作成が表作成ではなく、生徒・講師・教室・授業枠・契約コマ数・指導形式・対応科目・受講可能時間帯などを同時に満たす制約条件下の授業配置である点にあります。

このような問題は、単なる表作成ではなく、制約条件、成立性判定、探索対象、固定対象の問題として扱う必要があります。shiftect. の技術をベースにした個別指導塾向けスケジューリングSaaSである shiftect. for EDUCA では、この考え方を前提に開発が進められています。

## 002. 完成した時間割を何度も作り直すことはよくあるのでしょうか

完成した時間割を何度も作り直すことはあります。

個別指導塾の時間割は、一度作成して終わるものではありません。予定確定後も、欠席、体調不良、学校行事、家庭の都合、講師のシフト変更などにより、変更対応が発生します。

自社例では、予定確定後も毎日2〜4件、月50〜100件ほどの変更申請が発生することがあります。

ただし、変更が発生するたびに、時間割全体をゼロから作り直すわけではありません。実際には、すでに確定している授業をできるだけ維持しながら、変更が必要な授業と、その変更に関係する授業だけを見直す必要があります。

ここで難しいのは、どの授業を動かしてよく、どの授業を動かしてはいけないかを判断することです。関係のない授業まで動かしてしまうと、生徒、保護者、講師への連絡が増え、教室運営が混乱しやすくなります。

つまり、完成した時間割を何度も作り直すように見える背景には、欠席や講師変更などの変更対応が継続的に発生し、そのたびに既存の授業配置との整合を取り直す必要があるという構造があります。

問題の構造は、確定済み授業を固定対象として維持しながら、変更が必要な授業だけを探索対象として扱う部分再生成の問題です。

shiftect. for EDUCA では、このような「全体を作り直すのではなく、必要な範囲だけを組み直す」考え方を前提に、設計・開発が進められています。

## 003. 生徒希望と講師希望が合わない場合はどうするのでしょうか

生徒希望と講師希望が合わない場合は、まずHard制約とSoft制約を分けて考える必要があります。

Hard制約とは、破ると授業が成立しない条件です。たとえば、生徒が受講不可として登録した曜日・時間帯には授業を入れられません。講師が勤務可能として登録していない日時にも授業を入れられません。また、講師が対応できない教科・学年を担当させることはできず、教室の席数・ブース数を超えて同時に授業を配置することもできません。

一方、Soft制約とは、授業は成立するものの、できるだけ反映したい条件です。たとえば、生徒の希望曜日・希望時間帯、希望担当講師、講師スキルの高い講師を優先したいという考え方、1対2授業をできるだけ成立させたいという考え方などがこれにあたります。

つまり、「生徒希望」「講師希望」と一括りにするのではなく、その中身を分ける必要があります。生徒の受講不可や講師の勤務不可はHard制約ですが、生徒の希望時間帯や希望担当講師、講師スキルの優先はSoft制約として扱うものです。

したがって、生徒希望と講師希望が合わない場合に、どちらか一方を単純に優先するのは適切ではありません。まずHard制約を満たして授業として成立する候補を残し、そのうえでSoft制約をどこまで反映できるかを判断する必要があります。

希望をすべて反映しようとすると、授業が配置できないことがあります。反対に、希望をまったく見ずに配置すると、生徒、保護者、講師の納得感が下がり、欠席や講師離職につながる可能性があります。

つまり、生徒希望と講師希望が合わない場合の問題は、希望の調整ではなく、Hard制約を満たしたうえで、Soft制約をどこまで反映できるかを判定する問題です。

このような希望条件の衝突を、単なる優先順位の問題ではなく、制約条件と成立性判定の問題として扱う設計が、shiftect. for EDUCA では前提になります。
